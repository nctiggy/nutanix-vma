# Contributing to Nutanix VMA

## Development Setup

### Prerequisites

- Go 1.22+
- Docker (for building container images)
- kubectl with cluster access
- [golangci-lint](https://golangci-lint.run/usage/install/) v2.8+

### Clone and Build

```bash
git clone https://github.com/nctiggy/nutanix-vma.git
cd nutanix-vma
make build
```

### Run Tests

```bash
# Unit tests + integration tests (envtest)
make test

# Unit tests + integration tests separately
make test-integration

# Lint
make lint

# E2E tests (requires real Nutanix cluster)
NUTANIX_E2E=true NUTANIX_PC_URL=... make test-e2e
```

### Project Structure

```
cmd/mock-nutanix/main.go             # Standalone mock server

internal/
  controller/                        # Kubernetes controllers
    provider_controller.go           # NutanixProvider reconciler
    plan_controller.go               # MigrationPlan reconciler
    migration_controller.go          # Migration reconciler (state machine)
    hook_runner.go                   # Pre/post migration hook execution
  nutanix/                           # Prism Central/Element API client
  builder/                           # Nutanix VM -> KubeVirt VM translation
  transfer/                          # Disk transfer orchestration (CDI)
  validation/                        # Pre-flight validation engine
  observability/                     # Prometheus metrics
pkg/mock/                            # Mock Nutanix API server
test/
  integration/                       # envtest-based integration tests
  e2e/                               # Real cluster E2E tests
  fixtures/crds/                     # KubeVirt/CDI CRD stubs for envtest
config/
  crd/bases/                         # Generated CRD YAML (DO NOT EDIT)
  rbac/                              # Generated RBAC (DO NOT EDIT)
  samples/                           # Example CRs
```

## Making Changes

### After editing `*_types.go`

Always regenerate manifests and deepcopy methods:

```bash
make manifests && make generate
```

### After editing any `.go` file

```bash
make build && make test
```

### Code Style

- Run `make lint` before submitting. Zero lint issues required.
- Use standard `testing` for unit tests, Ginkgo v2 + Gomega for integration/E2E.
- Use `httptest` for mocking HTTP endpoints in unit tests.
- Always use `_ = resp.Body.Close()` to satisfy errcheck.
- Preallocate slices with `make([]T, 0, len)` to satisfy prealloc.

### Never Edit Auto-Generated Files

- `config/crd/bases/*.yaml` -- regenerated by `make manifests`
- `config/rbac/role.yaml` -- regenerated by `make manifests`
- `**/zz_generated.*.go` -- regenerated by `make generate`

### Never Remove Scaffold Markers

`// +kubebuilder:scaffold:*` comments are used by the kubebuilder CLI.

## Testing Conventions

- **Unit tests**: `*_test.go` files alongside source, use `httptest` for HTTP mocking
- **Integration tests**: `test/integration/`, use envtest (real K8s API + etcd) + mock Nutanix server
- **E2E tests**: `test/e2e/`, require real Nutanix cluster (gated behind `NUTANIX_E2E=true`)
- **Mock server**: `pkg/mock/` provides an in-process httptest server with configurable fixtures

## Pull Request Process

1. Fork the repository
2. Create a feature branch from `main`
3. Make your changes
4. Run the full check suite:
   ```bash
   make lint && make build && make test
   ```
5. Commit with a descriptive message
6. Push and open a Pull Request
7. CI will run lint, build, and tests automatically
8. Address any review feedback

### Commit Message Format

Use a short summary line followed by a body if needed:

```
Short summary of change (50 chars or less)

Longer description if needed. Wrap at 72 characters.
Explain *why* the change was made, not just *what* changed.
```

## Release Process

Releases are automated via GitHub Actions. To create a release:

1. Tag the commit: `git tag v0.1.0`
2. Push the tag: `git push origin v0.1.0`
3. The release workflow will:
   - Build and push the Docker image to `ghcr.io/nctiggy/nutanix-vma:v0.1.0`
   - Create a GitHub Release with auto-generated changelog
